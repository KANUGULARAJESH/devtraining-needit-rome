# Speaker Notes – SQLFluff Linting for dbt on Databricks

## Introduction

> "This slide explains how we implement SQLFluff as part of our dbt development lifecycle on Databricks. The objective is to enforce consistent SQL coding standards, detect issues early, and prevent poorly formatted SQL from reaching production."

---

## Step 1 – Install Required Packages

> "The first step is to install SQLFluff, the dbt templater, and the dbt-databricks adapter. The dbt templater allows SQLFluff to correctly understand dbt-specific Jinja syntax, while the Databricks adapter ensures SQL is parsed using the Databricks SQL dialect."

---

## Step 2 – Configure SQLFluff

> "Next, we configure the `.sqlfluff` file at the project root. Here we specify the dbt templater, Databricks dialect, project directory, profiles directory, and the SQL formatting and linting rules that will be enforced across the project."

---

## Step 3 – Ignore Generated Files

> "We configure a `.sqlfluffignore` file to exclude folders such as `target` and `dbt_packages`, since these are generated automatically by dbt. Optionally, macros can also be excluded if they are not intended for linting."

---

## Step 4 – Local Linting

> "Developers can run `sqlfluff lint` locally to identify SQL quality issues before committing their code. If formatting issues are detected, `sqlfluff fix` can automatically correct many of them, reducing manual effort."

---

## Step 5 – GitHub Actions Integration

> "When code is pushed or a Pull Request is created, GitHub Actions automatically executes SQLFluff. This ensures every code change is validated consistently without relying solely on local developer checks."

---

## Step 6 – Optional Pre-commit Hook

> "As an additional safeguard, developers can configure a pre-commit hook. This runs SQLFluff before the local Git commit is created, providing immediate feedback and preventing common formatting issues from entering the repository."

---

## Step 7 – CI Quality Gate

> "Finally, SQLFluff acts as a CI quality gate. If linting fails, the pipeline returns a non-zero exit code, the Pull Request is blocked, and the developer must resolve the issues before the code can proceed to the Slim CI build and merge."

---

## CI Pipeline (Right Side)

> "The flow on the right summarizes the process. A developer modifies a dbt model, optionally triggers a local pre-commit check, and pushes the changes. GitHub Actions runs SQLFluff automatically. If linting succeeds, the pipeline proceeds to the Slim CI build, where only the modified models and their downstream dependencies are built and tested. Once all validations pass, the Pull Request can be merged."

---

## Outcome

> "By integrating SQLFluff into our development workflow, we achieve standardized SQL formatting, automated Pull Request quality checks, early detection of SQL issues, and reduced code review effort. This helps improve code quality and maintain consistency across all dbt models."

---

### Expected client question

**Q: Why use SQLFluff if `dbt build` already runs in CI?**

**Answer:**

> "`dbt build` validates whether the models execute successfully, whereas SQLFluff checks coding standards, formatting, naming conventions, and SQL best practices before execution. Using both ensures we maintain high code quality as well as functional correctness."
