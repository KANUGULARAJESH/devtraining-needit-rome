## Speaker Notes

"This slide compares the two orchestration approaches we evaluated for running dbt on Databricks. While both options produce the same business output, they differ in where dbt executes and how the platform is managed.

### Option 1 – dbt packaged in a Databricks Asset Bundle (Recommended)

In this approach, Airflow is responsible only for orchestration. It triggers a Databricks Job using the DatabricksRunNowOperator, and the actual dbt execution happens entirely inside Databricks.

The main advantages are:

* Execution remains native to Databricks, so governance through Unity Catalog, lineage, and security stay within the Databricks platform.
* We package the dbt project as a Databricks Asset Bundle, giving us Infrastructure as Code and enabling consistent deployments across Development, SIT, and Production.
* Airflow remains lightweight because it only schedules and monitors jobs instead of executing dbt itself.
* We don't need to install or maintain dbt Core, adapters, profiles, or credentials on Airflow workers, reducing operational overhead.
* It also provides better control and visibility over Databricks job execution and compute utilization.

There are a few trade-offs:

* If a dbt model fails, engineers primarily troubleshoot from Databricks Job logs rather than directly in Airflow.
* We manage both the Databricks Asset Bundle deployment and the Airflow DAG.
* If model-level retries are required, they need to be designed as separate Databricks tasks rather than relying entirely on Airflow.

---

### Option 2 – Airflow runs dbt directly

In this approach, dbt Core is installed on the Airflow workers. Airflow executes dbt commands directly, while the SQL generated by dbt is submitted to Databricks SQL Warehouse for execution.

The advantages are:

* Airflow provides detailed task-level visibility, making it easier to monitor individual model executions.
* Failed models can be retried independently from the point of failure.
* Existing Airflow infrastructure can be reused without introducing Databricks Jobs for execution.

However, this approach has several operational drawbacks:

* Airflow workers now become responsible for maintaining dbt Core, adapters, profiles, credentials, and version compatibility.
* Governance responsibilities are split between Airflow and Databricks, making operations more complex.
* We now have two execution platforms to manage instead of one, increasing maintenance effort.
* Airflow worker utilization also increases because dbt runs directly on the workers.

---

### Recommendation

Although both approaches are technically valid, we recommend **Option 1** as the enterprise standard.

It keeps execution and governance native to Databricks, simplifies Airflow by limiting it to orchestration, enables repeatable deployments through Databricks Asset Bundles, and provides a more scalable and maintainable architecture for production workloads.

We would consider Option 2 only in scenarios where Airflow requires fine-grained control over individual dbt models or where an organization has a strong operational dependency on Airflow-based execution."

## One improvement to the slide

I'd also suggest changing the bottom line to make it more client-friendly:

> **Recommendation:** Adopt **Option 1** as the standard orchestration pattern for production workloads. It provides stronger governance, repeatable deployments through Databricks Asset Bundles, lower operational overhead, and better long-term scalability. Reserve **Option 2** for specialized use cases that require model-level orchestration within Airflow.

This wording is more aligned with architecture review presentations and clearly communicates the rationale behind the recommendation.
