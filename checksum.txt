# Speaker Notes – CI/CD Flow

## Introduction

> "This slide illustrates the end-to-end CI/CD process we propose for managing dbt transformations on Databricks. The objective is to ensure every code change is automatically validated, reviewed, and promoted through controlled environments before deployment to production."

---

## Step 1 – Open Pull Request

> "The process begins when a developer creates or modifies dbt models, tests, or macros in a feature branch. Once the changes are complete, a Pull Request is raised. This automatically triggers the CI pipeline, where an isolated CI schema is created so validation can be performed without impacting shared development environments."

---

## Step 2 – Quality & Validation

> "The first stage focuses on code quality. SQLFluff validates SQL formatting, naming conventions, and coding standards. We also validate the dbt project configuration. If any linting or validation fails, the CI pipeline stops immediately, preventing poor-quality code from progressing further."

---

## Step 3 – Slim CI Build

> "After the quality checks pass, we execute a Slim CI build. dbt compares the current project with the latest production `manifest.json` and identifies only the modified models along with their downstream dependencies. It then executes `dbt build --select state:modified+`, which builds only the impacted models and automatically runs their associated dbt tests. This significantly reduces CI execution time compared to a full project build."

---

## Step 4 – Review

> "Once the automated validations succeed, the Pull Request undergoes peer review. The reviewers validate the transformation logic, lineage impact, and CI results before approving the code for promotion."

---

## Step 5 – Merge

> "After approval, the Pull Request is merged into the integration branch. This generates a versioned deployment artifact and prepares the code for deployment into the next environment."

---

## Step 6 – Deploy

> "Finally, the deployment pipeline deploys the approved dbt project to the target Databricks environment. A production `dbt build` materializes the Silver and Gold models in Unity Catalog while publishing metadata, lineage, and execution logs for governance and traceability."

---

## Repository Versioning

> "We follow semantic versioning for the centralized dbt repository. Major versions represent breaking changes, minor versions introduce new features, and patch versions capture bug fixes. This provides controlled and traceable releases."

---

## Git Branching Strategy

> "Our Git strategy follows Feature → Development → SIT → Master. Every commit to a feature branch triggers the CI pipeline. Once the code passes validation and peer review, it is promoted sequentially through Development, SIT, and finally Production, ensuring controlled and governed releases."

---

## Closing

> "Overall, this CI/CD process combines automated quality checks, faster validation using Slim CI, controlled code reviews, versioned releases, and automated deployment, providing a reliable and scalable framework for managing dbt transformations on Databricks."

### One correction I'd recommend on your slide

In **Step 2**, replace:

> **"Validates dbt project compilation"**

with:

> **"Validates dbt project configuration"**

or

> **"Validates dbt project structure and dependencies"**

This is more accurate because the actual compilation happens during the `dbt build` in **Step 3**. This small change will make the slide technically stronger.
