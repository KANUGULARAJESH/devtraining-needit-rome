Here are speaker notes you can use during the presentation. This explanation is aimed at a technical client and should take about **3–4 minutes**.

---

# Introduction

> "This slide presents the two orchestration approaches we evaluated for executing dbt transformations on Databricks. Both approaches are technically feasible, but we recommend Option 1 because it provides better separation of responsibilities, easier operations, and improved scalability for enterprise deployments."

---

# CI/CD and Deployment (Top Section)

> "The process starts with the dbt project stored in Git. Whenever code is pushed or a Pull Request is created, the CI/CD pipeline performs code validation and deployment. Using Databricks Asset Bundles (DAB), the same Databricks Job definition is promoted consistently across Development, Staging, and Production environments."

---

# Option 1 – Recommended Architecture

> "In the recommended approach, Airflow is responsible only for orchestration. It schedules the workflow and triggers a pre-deployed Databricks Job using the DatabricksRunNowOperator."

> "The actual dbt execution happens inside a dedicated Databricks Job Cluster. This compute runs the dbt CLI commands such as `dbt deps` and `dbt build`, compiles the models, resolves dependencies, and submits the generated SQL to the Databricks SQL Warehouse through the dbt-databricks adapter."

> "The SQL Warehouse executes the SQL and reads or writes data managed in Unity Catalog."

### Why we recommend this approach

> "This architecture cleanly separates orchestration from execution. Airflow remains lightweight because it doesn't need to manage dbt installations or dependencies."

> "Since Databricks Asset Bundles package both the dbt project and the Databricks Job definition, the same deployment pattern is consistently promoted across Development, Staging, and Production."

> "Because the execution environment is fully managed inside Databricks, cluster configuration, libraries, authentication, and scaling are centrally managed, reducing operational overhead."

> "This approach also improves reliability, since dbt executes using dedicated Databricks compute optimized for transformation workloads."

---

# Data Flow

> "The transformations follow the Medallion architecture. Bronze contains the raw ingested data. Databricks performs the initial Bronze-to-Silver transformations, while dbt is responsible for creating curated Silver and business-ready Gold models. Unity Catalog governs access, metadata, and lineage across these layers."

---

# Option 2 – Alternative Approach

> "The second approach executes dbt directly on the Airflow workers using BashOperator or DbtTaskGroup."

> "In this case, Airflow must host dbt Core, the dbt-databricks adapter, all required Python packages, and manage authentication to Databricks."

> "The Airflow worker compiles the dbt project and submits the compiled SQL to the Databricks SQL Warehouse for execution."

### Challenges

> "Although this approach works, it introduces additional operational responsibilities. Every Airflow worker must maintain the correct dbt version, adapters, and dependencies."

> "Any upgrade to dbt or supporting libraries must be coordinated across the Airflow environment, increasing maintenance effort."

> "Scaling also becomes more difficult because Airflow is responsible for both orchestration and execution."

> "For these reasons, this approach is generally better suited for smaller implementations or organizations that already have a mature Airflow-based dbt platform."

---

# Final Recommendation

> "For the Holcim implementation, we recommend Option 1. It aligns with Databricks best practices by keeping orchestration in Airflow while delegating execution to Databricks Jobs. This provides better scalability, simpler operations, environment consistency through Databricks Asset Bundles, and easier long-term maintenance."

---

## Questions you may be asked

**Q: Why are there two computes in Option 1?**

> "Airflow is not a compute engine; it only orchestrates workflows. The Databricks Job Cluster provides the compute required to execute dbt commands. The SQL Warehouse then executes the SQL generated by dbt. Separating orchestration, dbt execution, and SQL execution allows each component to scale independently."

**Q: Why not execute dbt directly on Airflow?**

> "Running dbt on Airflow requires managing dbt installations, adapters, libraries, and authentication on every Airflow worker. Using Databricks Jobs shifts this responsibility to Databricks, making the platform easier to operate and scale."
